# 블로그 LLM 위키 운영 스키마

## 목적

이 저장소는 단순한 Jekyll 블로그가 아니라, LLM이 지속적으로 유지보수하는 개인 지식 위키를 기반으로 글을 발행하는 저장소다. 새 소스가 들어오면 바로 `_posts/`에 쓰지 않고, 먼저 위키에 흡수하고 정리한 뒤 공개 글로 승격한다.

## 저장소 계층

### 1. 원천 소스 계층: `raw/`

- 사람이 수집한 원문 자료를 보관한다.
- 이 계층은 사실상 불변이다. 정리용 파일명 변경이나 폴더 이동은 가능하지만, 원문 내용을 요약본처럼 덮어쓰지 않는다.
- 새 자료는 먼저 `raw/inbox/`에 두고, 검토 후 `raw/sources/`로 정리한다.
- 원문 이미지와 첨부 파일은 `raw/assets/`에 둔다.

### 2. 지식 위키 계층: `wiki/`

- LLM이 읽고 쓰는 핵심 작업 공간이다.
- 원문 요약, 개념 페이지, 시리즈 페이지, 질의응답 기록, 발행 후보 초안을 모두 이 계층에서 관리한다.
- 핵심 엔트리 파일은 아래와 같다.
  - `wiki/index.md`: 위키 전체 카탈로그
  - `wiki/log.md`: ingest, query, lint, publish 이력
  - `wiki/overview.md`: 블로그 주제 범위와 운영 관점

### 3. 공개 발행 계층: `_posts/`

- 외부에 공개하는 최종 글만 둔다.
- 가능하면 `wiki/drafts/`나 관련 `wiki/topics/` 페이지를 바탕으로 작성한다.
- 원천 소스만 읽고 바로 `_posts/`를 쓰는 흐름은 예외로 취급한다.

### 4. 사이트 프레젠테이션 계층

- `_pages/`, `_data/`, `assets/`는 공개 사이트 UI와 탐색 구조를 담당한다.
- 내부 운영 문서와 지식 위키는 공개 사이트에 노출하지 않는다.

## 디렉터리 규칙

### `raw/`

- `raw/inbox/`: 아직 분류하지 않은 신규 소스
- `raw/sources/`: 장기 보관할 원천 자료
- `raw/assets/`: 이미지, PDF 보조 파일, 캡처

### `wiki/`

- `wiki/sources/`: 원천 자료별 요약 페이지
- `wiki/topics/`: 개념, 사람, 도구, 주장, 비교 대상 페이지
- `wiki/series/`: 반복 발행할 시리즈나 콘텐츠 축
- `wiki/queries/`: 질문에 대한 답변 중 재사용 가치가 있는 페이지
- `wiki/drafts/`: 공개 글 후보 초안
- `wiki/_templates/`: 새 페이지를 만들 때 따를 템플릿

## 운영 워크플로

### Ingest

1. 새 자료를 `raw/inbox/`에 추가한다.
2. 원문을 읽고 `wiki/sources/`에 소스 요약 페이지를 만든다.
3. 관련 `wiki/topics/`, `wiki/series/`, `wiki/drafts/`를 갱신한다.
4. `wiki/index.md`와 `wiki/log.md`를 함께 갱신한다.

### Query

1. 먼저 `wiki/index.md`를 읽어 관련 페이지를 찾는다.
2. 필요한 위키 페이지를 읽고 답변을 구성한다.
3. 재사용 가치가 있는 답변은 `wiki/queries/` 또는 `wiki/drafts/`에 파일링한다.

### Lint

정기적으로 아래 항목을 점검한다.

- 최신 소스로 뒤집힌 주장
- 서로 충돌하는 서술
- 링크되지 않은 고아 페이지
- 반복 언급되지만 페이지가 없는 개념
- 공개 글로 승격할 만한 위키 초안

### Publish

1. `wiki/drafts/`에서 공개 가치가 있는 초안을 고른다.
2. 표현과 구조를 블로그 글 형식으로 다듬는다.
3. 최종 결과만 `_posts/`에 반영한다.
4. 발행 후 관련 위키 페이지에 게시 사실을 반영한다.

## 작성 규칙

- 기본 언어는 한국어다.
- 위키 문서는 가능한 한 상대 링크로 연결한다.
- 지속 가치가 있는 분석은 채팅에만 남기지 말고 파일로 남긴다.
- 소스 요약에는 원문 경로 또는 URL을 반드시 기록한다.
- 위키 페이지에는 가능하면 YAML front matter를 두고 `title`, `type`, `status`, `updated_at`, `tags`를 명시한다.
- `wiki/log.md`의 항목 헤더는 `## [YYYY-MM-DD] action | 제목` 형식을 유지한다.

## 블로그 글 작성 규칙

- `_posts/`와 `wiki/drafts/`의 기술 글은 `docs/technical-post-writing-guide.md`와 `docs/blog-engineering-guide.md`를 함께 따른다.
- 기술 글은 단순 요약보다 사실의 재구성과 맥락 연결을 우선한다.
- 기본 문체는 친근한 존댓말 `~요`체로 유지하되, 과장이나 홍보성 표현은 피한다.
- 외부 소스를 직접 인용하는 글은 위 가이드의 본문 주석과 `## 출처` 형식을 따른다.

## 코드와 테스트 변경 규칙

- 모든 공개 함수와 메서드는 한국어 docstring을 가진다.
- 비자명 로직에는 판단 근거가 드러나는 한국어 주석을 남긴다.
- 테스트 코드도 같은 문서화 규칙을 따른다.
- 커밋 메시지는 한국어 한 줄로 작성하고, `동사:이유 + 내역` 형식을 따른다.
