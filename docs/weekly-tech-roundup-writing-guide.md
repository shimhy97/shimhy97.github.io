# 주간 기술동향 글 작성 규약

## 문서 목적

이 문서는 `_posts/*-weekly-tech-roundup.md` 계열 글이 자동 요약문처럼 보이지 않도록, 주간 기술동향 글의 어투, 구조, 편집 원칙을 고정한다. 목적은 "정보를 많이 담는 것"이 아니라 "운영자가 실제로 읽고 싶고, 구독자가 사람 글로 받아들이는 글"을 만드는 것이다.

이 규약은 LLM 초안, 사람 편집, 이후 렌더러 개편의 공통 기준으로 사용한다.

## 왜 지금 필요한가

2026-04-03자 주간 글은 아래 이유로 AI가 쓴 초안처럼 읽힌다.

- 모든 항목이 같은 3문장 골격으로 반복된다.
- 도입부에 글쓴이의 관찰이나 판단이 없다.
- "보안이나 호환성 리스크", "로드맵과 함께 판단"처럼 추상 문장이 자주 반복된다.
- 릴리즈 노트의 문장을 거의 그대로 옮긴 부분은 있는데, 왜 그 항목을 골랐는지는 보이지 않는다.
- 마무리가 글의 해석이 아니라 자동화 파이프라인 설명으로 끝난다.

좋은 기술 글은 대체로 "자료 정리"보다 "관찰 메모"에 가깝다. 같은 사실을 다뤄도, 무엇이 눈에 띄었고 왜 중요하다고 봤는지가 먼저 드러난다.

## 조사에서 반복된 공통점

### 1. 첫 문단에서 이미 관점이 드러난다

좋은 글은 "이번 주에는 여러 소식이 있었다"로 시작하지 않는다. 대신 아래 셋 중 둘 이상을 첫 문단에 바로 넣는다.

- 무엇이 바뀌었는가
- 왜 지금 봐야 하는가
- 누가 영향을 받는가

### 2. 추상어보다 증거가 먼저 나온다

사람이 쓴 글처럼 보이는 문장은 대체로 구체 명사와 수치를 먼저 건다.

- 버전
- 날짜
- 마이그레이션 시점
- 장애 범위
- 내부 도구명
- 실패 조건
- 코드, 스크린샷, 명령어, 인용

### 3. 사실과 해석이 분리되어 있다

좋은 기술 글은 사실을 적고 끝내지 않는다. 그렇다고 해석만 길게 끌지도 않는다. 대부분 아래 흐름을 가진다.

1. 사실 한 줄
2. 그 사실이 중요한 이유
3. 독자가 취할 행동 또는 아직 남은 불확실성

### 4. 문단 길이와 리듬이 일정하지 않다

실제 사람이 쓴 글은 짧은 단정문, 긴 배경 설명, 불릿, 인용, 소제목이 섞인다. 반대로 AI 냄새가 나는 글은 모든 항목이 같은 길이와 같은 톤으로 반복된다.

### 5. 약한 주관이 들어간다

완벽한 중립보다 제한된 판단이 더 자연스럽다.

- "이번 주엔 AI 발표보다 인프라 운영 쪽 변화가 더 실질적으로 보였다."
- "흥미로운 점은 기능 자체보다 전환 시점을 못 박았다는 점이다."
- "당장 도입감보다, 기존 설정을 건드릴 가능성이 더 크다."

이런 문장은 과한 의견이 아니라 편집자의 시선이다.

### 6. 결론은 요약이 아니라 다음 행동으로 끝난다

좋은 기술 글은 "앞으로도 주목할 만하다"로 닫지 않는다. 대신 아래 중 하나로 닫는다.

- 당장 확인할 설정
- 다음 주에 볼 지표
- 아직 불확실한 부분
- 도입 전에 검증할 포인트

## 주간 기술동향 글의 기본 관점

주간 기술동향 글은 뉴스 모음집이 아니라 "이번 주 개발 생태계에서 무엇이 실제 신호였는지 정리한 편집 메모"여야 한다.

따라서 공개 글에는 아래가 필요하다.

- 이번 주를 한 줄로 요약하는 관찰
- 항목을 고른 이유
- 원문 요약과 별개인 편집자의 해석
- 당장 확인할 실무 포인트

반대로 공개 글에서 아래는 줄인다.

- 파이프라인 설명
- 과도한 선정 기준 나열
- 모든 항목에 똑같은 문장 뼈대 적용
- 누구에게나 적용되는 추상 조언

## 권장 글 구조

### 1. 도입부

도입부는 2~4문장으로 쓴다.

- 첫 문장에서 이번 주의 핵심 흐름을 단정한다.
- 둘째 문장부터 왜 그렇게 봤는지 구체 항목 2~3개를 언급한다.
- 선정 기준은 본문 상단에서 길게 설명하지 않는다.
- 자동 생성 여부는 도입부와 마무리에서 숨긴다.

좋은 예시:

```text
이번 주엔 새 기능 발표보다 "기존 운영 환경을 다시 점검하게 만드는 변화"가 많았다. GitHub Actions 보안 옵션, Node.js 보안 운영 공지, Next.js 패치 릴리즈처럼 당장 기능 도입보다 운영 체크리스트를 건드리는 항목이 더 눈에 띄었다.
```

피해야 할 예시:

```text
이번 주에는 클라우드 / DevTools / 플랫폼, OSS 릴리즈 / 보안 영역의 공식 업데이트가 두드러졌습니다. 아래 다이제스트는 7일 내 공개된 공식 changelog, engineering blog, GitHub Releases를 기준으로 선별했습니다.
```

### 2. 섹션 구성

섹션은 카테고리별로 기계적으로만 나누지 말고, 그 주의 관찰이 보이도록 이름을 붙인다.

- "배포 파이프라인을 건드리는 변화"
- "지금 바로 읽어둘 만한 보안 신호"
- "새 기능보다 마이그레이션 포인트가 중요한 릴리즈"

섹션마다 1~2문장 짧은 도입을 붙인다. 이 문장이 있어야 목록이 아니라 글처럼 읽힌다.

### 3. 항목 단위

항목 하나는 "사실 문장 + 해석 문장 + 확인 포인트"의 2~4문장으로 쓴다. 내부 데이터 모델에서 `what_changed`, `why_it_matters`, `practical_impact`를 유지해도 되지만, 공개 글에서는 그 필드명을 그대로 노출하지 않는다.

권장 형식:

```markdown
### GitHub Actions: Early April 2026 updates

GitHub Actions가 2026년 4월 초 업데이트에서 서비스 컨테이너 entrypoint override와 OIDC custom properties를 추가했다. 이번 변화는 새 기능 자체보다, 기존 CI 보안 구성과 네트워크 격리 전략을 다시 읽게 만든다는 점에서 더 중요하다. 특히 OIDC를 이미 쓰는 팀이라면 현재 claim 설계와 self-hosted runner 구성을 같이 점검해볼 만하다.

- 원문: ...
```

보조 줄은 아래 두 개까지만 허용한다.

- `- 원문: ...`
- `- 체크 포인트: ...`

출처, 발행일, 링크, 핵심 요약을 매번 별도 불릿 4개로 반복하지 않는다.

### 4. 마무리

마무리는 파이프라인 설명이 아니라 편집 메모로 끝낸다.

- 이번 주에 실제로 중요했던 흐름 1개
- 다음 주에 이어서 볼 포인트 1~2개
- 아직 판단을 유보해야 하는 영역 1개

예시:

```text
이번 주 글을 묶어보면, 새 제품 소개보다 기존 운영 표면을 다시 읽게 만드는 변경이 많았다. 다음 주엔 실제로 deprecation 날짜가 붙는 항목이 더 나오는지, 그리고 GitHub Actions와 Next.js 쪽 변경이 운영 가이드까지 이어지는지 계속 볼 생각이다.
```

## 문장 규약

### 반드시 넣을 것

- 항목마다 구체 명사 2개 이상
- 항목마다 날짜, 버전, 지표, 기능명 중 최소 1개
- 글 전체에서 편집자의 판단이 드러나는 문장 3개 이상
- 섹션 도입 또는 마무리에서 다음 행동 제안

### 자주 써도 되는 표현

- "이번 주엔 ... 쪽이 더 눈에 띄었다."
- "흥미로운 점은 ... 보다 ... 에 있다."
- "실무적으로는 ... 부터 확인하는 편이 낫다."
- "당장 도입할 기능이라기보다 ... 신호에 가깝다."
- "이 항목은 발표 자체보다 ... 시점을 봐야 한다."

### 금지 표현

- "공지의 핵심은 ..."
- "왜 중요한가:"
- "실무 영향:"
- "공식 제품 채널에서 나온 변화라 ..."
- "보안이나 호환성 리스크로 이어질 수 있어 ..."
- "로드맵과 함께 판단하면 됩니다."
- "앞으로도 주목할 만하다."
- "자동 생성 초안입니다."

금지 이유는 간단하다. 이런 문장은 거의 어디에나 붙을 수 있어서, 해당 항목을 실제로 읽은 흔적이 사라진다.

## 사람 냄새를 만드는 장치

아래 장치 중 최소 3개 이상이 글 전체에 보여야 한다.

- 의외성: "생각보다 새 기능보다 폐기 일정이 더 중요했다."
- 긴장: "좋은 소식이라기보다 운영팀이 다시 확인할 공지에 가깝다."
- 한계 인정: "아직 preview 단계라 바로 일반화하긴 이르다."
- 비교: "이번 변화는 지난 릴리즈보다 운영 측면 영향이 크다."
- 구체 디테일: 버전, 팀명, 내부 도구명, 실패 조건, 장애 시간
- 자기 위치: "내 기준에선", "운영 관점에선", "프론트엔드 팀이라면"

단, 과장된 감탄이나 억지 캐릭터 문체는 금지한다. 사람 냄새는 과한 개성보다 구체성과 판단에서 나온다.

## 권장 작성 프로세스

### 1. 후보를 읽고 사실 메모만 남긴다

초안 1단계에서는 문장을 예쁘게 만들지 않는다. 아래만 뽑는다.

- 무엇이 바뀌었는가
- 언제 공개되었는가
- 누가 영향 받는가
- 어떤 숫자나 제약이 있는가

### 2. 이번 주 관찰 한 줄을 먼저 정한다

항목을 다 모은 뒤, 이번 주 글을 관통하는 판단을 한 줄로 정한다. 이 문장이 도입부와 섹션 제목을 결정한다.

### 3. 공개 글에서는 데이터 필드를 문단으로 재구성한다

내부 요약 스키마는 유지해도 되지만, 공개 글에서는 아래 순서로 다시 쓴다.

1. 사실 문장
2. 해석 문장
3. 체크 포인트 또는 원문 링크

### 4. 마지막 편집에서 반복문을 걷어낸다

최종 검토에서는 아래를 확인한다.

- 같은 문장형이 2회 이상 반복되는가
- 모든 항목이 같은 길이인가
- 특정 항목에만 있는 구체성이 다른 항목엔 빠졌는가
- 도입부와 마무리에 편집자의 판단이 보이는가

## PR 검토 체크리스트

- 첫 문장에서 이번 주의 관찰이 드러나는가
- 섹션 제목이 카테고리명 복붙이 아니라 편집 관찰을 담고 있는가
- 각 항목에 실제 디테일이 있는가
- 각 항목의 해석이 서로 다른가
- 똑같은 경고 문장이 반복되지 않는가
- 마무리가 자동화 설명이 아니라 다음 관찰 포인트로 끝나는가

## 후속 구현 메모

현재 렌더러는 공개 포스트를 `무슨 변화인가 / 왜 중요한가 / 실무 영향` 불릿으로 그대로 출력한다. 이 구조는 내부 데이터 스키마로는 유효하지만, 공개 글 구조로는 부자연스럽다.

다음 구현 단계에서는 아래를 목표로 둔다.

- 요약 필드는 유지하되 렌더러에서 서술형 문단으로 재구성한다.
- 섹션 도입문과 마무리 문단에 편집 관찰을 넣는다.
- 동일한 경고 문장 재사용을 줄이기 위해 항목별 근거 중심 문장을 우선 생성한다.

이 문서를 기준으로 이후 프롬프트, 렌더러, PR 검토 기준을 함께 조정한다.
