# 기술 글 작성 가이드

## 문서 목적

이 문서는 `_posts/`와 `wiki/drafts/`에 들어가는 기술 글이 단순 요약문이나 AI 초안처럼 보이지 않도록, 제목, 구조, 문체, 근거 표기 기준을 고정한다. 목표는 "정보를 많이 넣는 글"이 아니라 "사실을 바탕으로 맥락을 다시 엮는 글"을 만드는 것이다.

화면 밀도, 요약 박스, 시각 요소, 본문 폭 같은 게시 형식 규칙은 `docs/blog-engineering-guide.md`를 함께 따른다. 이 문서는 어디까지나 "무엇을 어떻게 쓸 것인가"에 집중한다.

## 적용 범위

- 저장소 구현기, 운영 회고, 아키텍처 해설, 도구 비교, 공식 문서 해설 같은 일반 기술 글
- `wiki/drafts/`에서 `_posts/`로 승격하는 발행 후보 초안
- 여러 외부 소스와 내부 위키를 함께 엮어 쓰는 해설형 글

주간 기술동향 글도 별도 예외 없이 이 문서를 기본 양식으로 사용한다.

## 기본 관점

기술 글은 자료를 나열하는 문서가 아니라, "이번에 무엇이 달라졌고 왜 같이 봐야 하는지"를 설명하는 편집 결과여야 한다.

- 핵심은 사실의 재구성이다. 기사, 문서, 코드, 운영 기록을 따로 요약하지 말고 하나의 흐름으로 엮는다.
- 중심축을 먼저 정한다. 글을 쓰기 전에 이번 글이 독자에게 남길 한 문장을 먼저 고정한다.
- 사실과 해석을 분리한다. 확인된 정보와 필자의 판단을 같은 문장 안에 섞더라도, 어디까지가 근거인지 흐리지 않는다.
- 입력 데이터에 없는 사실을 추가하지 않는다. 추측, 과장, 홍보성 표현, 과도한 단정은 피한다.
- 다중 소스 글은 대표 소스와 보충 소스의 역할을 나눈다. 대표 소스가 글의 뼈대가 되고, 보충 소스는 배경, 비교, 반례, 후속 맥락을 채운다.
- 단일 소스 글이라도 저장소 내부 문서, 설계 의도, 테스트, incident 기록과 연결해 맥락을 만든다.

## 문체와 톤

- 기본 문체는 친근한 존댓말의 `~요`체로 쓴다.
- 보고서처럼 지나치게 딱딱하거나, 홍보 문안처럼 과장된 문장은 피한다.
- 실제 편집자가 독자에게 설명하듯, 건조하지만 읽기 쉽게 쓴다.
- 같은 문장 길이와 같은 어미를 반복하지 않는다.
- "주목", "핵심", "인사이트", "트렌드", "시사점", "본격화", "고도화" 같은 상투어는 습관적으로 쓰지 않는다.

## 권장 글 구조

기술 글은 아래 양식을 기본 골격으로 삼는다.

1. 제목 1개
2. 오프닝 3~5문장
3. 본문 섹션 4~6개
4. 마무리 2~4문장
5. 외부 소스를 인용한 경우 `## 출처`

짧은 회고나 노트형 글은 섹션 수를 줄일 수 있지만, 최소한 아래 흐름은 유지한다.

- 무엇이 달라졌는가
- 왜 지금 같이 봐야 하는가
- 어디에서 변화가 드러나는가
- 독자가 확인할 포인트가 무엇인가

## 제목과 소제목 규칙

- 제목은 뉴스 기사 제목처럼 짧고 구체적으로 쓴다.
- 제목은 멋을 내기보다, 독자가 글의 주제와 범위를 바로 알 수 있게 만든다.
- 소제목은 명사 나열보다 사건, 변화, 비교가 드러나는 문장형으로 쓴다.
- 제목 1개와 소제목 4~6개를 먼저 만든 뒤, 표현이나 어미가 겹치면 고쳐서 본문을 쓴다.
- 소제목 길이는 12~28자 정도를 권장한다.
- 추상어보다 고유명사, 날짜, 버전, 기능명, 장애명, 행동을 우선한다.

## 문단 구성 규칙

각 섹션은 목록 나열보다 짧은 서술 단락을 우선한다. 한 단락 안에서 아래 요소를 자연스럽게 섞는다.

1. 확인된 사실
2. 그 사실이 중요한 이유
3. 앞뒤 맥락과의 연결
4. 독자가 확인할 포인트 또는 남은 불확실성

대표 소스 1개만 반복 설명하지 말고, 보충 소스가 있다면 "왜 이 자료를 함께 봐야 하는지"를 붙인다.

## 용어와 식별자 설명 규칙

- 약어는 처음 등장할 때 풀어쓴 이름과 한국어 설명을 함께 적는다.
- 영어 약어는 가능하면 `한국어 설명 (영문 풀네임, 약어)` 순서로 적는다.
- 저장소 내부 전략 ID, 클래스명, 설정 키 같은 개발 식별자는 처음 등장할 때 역할을 먼저 설명하고, 코드 식별자는 뒤에 붙인다.
- 같은 글 안에서 한 번 설명한 약어나 식별자는 그다음부터 약식으로 써도 된다.

## 수식 표기 규칙

- 이 블로그의 수식 렌더링은 MathJax를 기준으로 한다.
- 블록 수식은 `$$ ... $$`를 기본으로 사용한다.
- inline 수식 중 첨자 `_`, 위첨자 `^`, 중괄호가 들어가는 경우는 `<span markdown="0">\(...\)</span>` 패턴을 기본으로 사용한다.
- 단순한 inline 수식만 `$...$`를 허용한다. 예를 들어 `$x$` 정도는 가능하지만, `$\hat{y}_{ij}$`처럼 복잡한 식은 피한다.
- Kramdown이 inline `$...$` 안의 `_`를 강조 문법으로 오인해 HTML이 깨질 수 있으므로, 복잡한 inline 수식은 반드시 위 패턴을 우선한다.
- 수식이 포함된 글은 가능하면 `rbenv exec bundle exec jekyll build`로 생성 HTML을 한 번 확인하고 발행한다.

예시:

- `시가 구간 돌파 전략 (Opening Range Breakout, ORB)`
- `상대거래량 (Relative Volume, RVOL)`
- `5분 이동평균 교차 전략인 \`sma_cross_5m\``
- `리스크 엔진 \`RiskEngine\``
- `시간대별 전략 전환 설정인 \`session_schedule\``

### 예시 관점

- 새 기능 발표 자체보다, 기존 운영 체크리스트를 다시 읽게 만드는 변화인가
- 기능 추가보다 deprecation 날짜나 마이그레이션 경로가 더 중요한가
- 코드 구조보다 incident와 테스트가 이 저장소의 실제 품질 기준을 보여주는가
- 하나의 도구 소개가 아니라, 팀의 운영 방식 변화로 읽어야 하는가

## 근거와 출처 표기

외부 기사, 공식 문서, 릴리즈 노트, 외부 저장소를 직접 인용한 글은 아래 형식을 따른다.

- 본문 인용은 반드시 `<sup><a href="#src-n">[n]</a></sup>` 형식을 사용한다.
- 문서 마지막에 반드시 `## 출처` 섹션을 둔다.
- 출처 항목은 `[n]`으로 시작하고, 각 항목에 `src-n` 앵커를 포함한다.
- 출처는 한 줄에 하나씩 쓰고, 항목 사이에는 빈 줄 1개를 둔다.
- URL은 그대로 노출하지 말고, 반드시 `[원문보기]` 텍스트에 링크를 건다.

예시:

```markdown
<span id="src-1"></span>[1] 매체명, 기사제목, <a href="https://example.com/article1" target="_blank" rel="noopener noreferrer">[원문보기]</a>
```

저장소 내부 코드, 위키 문서, 운영 메모를 바탕으로 쓰는 글은 외부 출처 섹션이 필수는 아니다. 대신 아래를 지킨다.

- 관련 파일, 문서, incident 이름을 본문에 구체적으로 적는다.
- 단순 체험담으로 흐르지 않도록, 코드 경로, 테스트 범위, 운영 시점 같은 식별 가능한 단서를 남긴다.
- 외부 소스와 내부 경험을 함께 쓴 글이라면 외부 자료만이라도 `## 출처`에 정리한다.

## 반드시 넣을 것

- 오프닝 첫 문단에서 이번 글의 중심 관찰
- 글 전체에서 구체 명사, 날짜, 버전, 기능명, 파일명 같은 식별 가능한 디테일
- 편집자의 제한된 판단이 드러나는 문장
- 마지막 문단의 확인 포인트, 남은 과제, 다음 관찰 지점

## 피할 것

- 기사나 문서별 요약을 같은 길이로 반복하는 구성
- "왜 중요한가", "실무 영향" 같은 필드명을 본문에 그대로 노출하는 방식
- 어떤 글에도 붙일 수 있는 추상 문장
- 근거 없는 전망, 홍보성 수사, 감탄 위주의 문장
- 자동화 파이프라인 설명으로 마무리하는 결론

## 권장 작성 절차

### 1. 먼저 사실 메모만 남긴다

초안 첫 단계에서는 예쁜 문장을 만들지 말고 아래만 정리한다.

- 무엇이 바뀌었는가
- 언제, 어디에서 확인했는가
- 누가 영향을 받는가
- 어떤 제약, 숫자, 실패 조건이 있는가

### 2. 글의 중심축을 한 줄로 정한다

대표 제목을 쓰기 전에 "이번 글은 결국 무엇을 말하는가"를 한 문장으로 고정한다. 오프닝 첫 문장은 이 중심축을 거의 그대로 드러내는 편이 낫다.

### 3. 소제목을 먼저 설계한다

본문을 쓰기 전에 제목 1개와 소제목 4~6개를 먼저 만든다. 소제목끼리 표현이 겹치거나 같은 리듬으로 끝나면 다시 고친다.

### 4. 문단을 요약이 아니라 연결로 쓴다

각 문단은 개별 자료 요약으로 끝내지 말고, 바로 앞 문단과 무엇이 이어지는지 함께 적는다. 독자가 "그래서 이 자료를 왜 같이 봐야 하는지"를 따로 추론하지 않아도 되게 만든다.

### 5. 마지막에 반복문을 걷어낸다

최종 검토에서는 아래를 본다.

- 같은 문장형이 두 번 이상 반복되는가
- 같은 추상어가 연속으로 나오지는 않는가
- 제목과 소제목이 너무 비슷하지 않은가
- 마무리가 요약 반복으로 끝나지 않는가

## 운영 원칙으로 요약하면

좋은 기술 글은 정보량보다 편집 밀도로 판단한다. 자료를 많이 모으는 것보다, 어떤 사실을 중심으로 묶고 무엇을 덜어냈는지가 더 중요하다.
