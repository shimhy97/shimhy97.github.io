# 블로그 LLM 위키 Workflow

## 문서 목적

이 문서는 `docs/llm-wiki-ENG.md`의 철학을 이 저장소에 어떻게 구체화했는지 설명한다. 블로그를 바로 글 저장소로만 쓰지 않고, 원천 자료와 공개 글 사이에 LLM 유지보수형 위키 계층을 둔다.

## 저장소 구조

### 1. 원천 자료

- 위치: `raw/`
- 역할: 사람이 수집한 원문을 장기 보관한다.
- 특징: 원문은 사실상 불변이며, 요약이나 해석은 여기 넣지 않는다.

### 2. LLM 위키

- 위치: `wiki/`
- 역할: 소스 요약, 주제 정리, 질의 기록, 발행 초안을 유지한다.
- 특징: LLM이 중심이 되어 갱신하며, `index.md`와 `log.md`를 함께 관리한다.

### 3. 공개 글

- 위치: `_posts/`
- 역할: 외부 독자에게 공개하는 최종 글을 담는다.
- 특징: 위키에서 검토된 해석과 구조를 바탕으로 작성하며, 기술 글 양식은 `docs/technical-post-writing-guide.md`를 따른다.

## 운영 흐름

### 1. Ingest

1. 새 자료를 `raw/inbox/`에 넣는다.
2. 보관 가치가 확인되면 `raw/sources/`로 정리한다.
3. `wiki/sources/`에 요약 페이지를 만든다.
4. 관련 `wiki/topics/`, `wiki/series/`를 갱신한다.
5. `wiki/index.md`와 `wiki/log.md`를 갱신한다.

### 2. Query

1. 질문이 들어오면 먼저 `wiki/index.md`를 확인한다.
2. 관련 페이지를 읽고 답변을 합성한다.
3. 일회성으로 끝내기 아까운 답변은 `wiki/queries/`에 남긴다.

### 3. Lint

정기적으로 아래를 점검한다.

- 최신 소스와 충돌하는 오래된 주장
- 링크되지 않은 페이지
- 자주 언급되지만 아직 없는 개념 페이지
- 공개 글로 승격할 만한 초안

### 4. Publish

1. `wiki/drafts/` 또는 `wiki/topics/`에서 공개 가치가 높은 초안을 고른다.
2. 독자 관점에 맞춰 구조를 다듬고, 기술 글이라면 `docs/technical-post-writing-guide.md`의 제목, 오프닝, 섹션, 마무리, 출처 규칙을 확인한다.
3. 최종 글을 `_posts/`에 반영한다.
4. 발행 후 관련 위키 페이지에 링크와 상태를 갱신한다.

## 운영 체크리스트

### 새 소스 추가 시

- 원문 경로나 URL이 남아 있는가
- 이미 다룬 주제와 어떤 연결점이 있는가
- 기존 주장과 충돌하는 내용이 있는가
- 블로그 발행 후보로 확장할 가치가 있는가

### 글 발행 전

- 위키에서 근거 페이지가 정리되어 있는가
- 단순 요약이 아니라 해석 또는 비교가 포함되는가
- 공개 글과 내부 위키의 역할이 섞이지 않았는가
- Jekyll 빌드가 통과하는가
