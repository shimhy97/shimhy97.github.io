# Feature Specification: Weekly Tech Digest Agent

**Feature Branch**: `[001-weekly-tech-digest-agent]`  
**Created**: 2026-04-03  
**Status**: Draft  
**Input**: User description: "최신 IT 개발 기술을 수집하여 주 1회 한국어 요약 글로 정리하고 GitHub Pages 블로그에 배포하는 에이전트와 운영 방식을 정의한다."

## User Scenarios & Testing

### User Story 1 - 공식 소스를 등록하고 주간 후보를 안정적으로 수집한다 (Priority: P1)

블로그 운영자는 최신 기술 소식이 흩어져 있어도, 신뢰 가능한 공식 소스 목록만 관리하면 최근 7일 기준 후보 항목을 자동으로 모을 수 있어야 한다.

**Why this priority**: 주간 글의 품질은 수집 원천의 품질에 직접 좌우된다. 공식 소스를 안정적으로 수집하지 못하면 이후 요약과 발행 단계가 모두 흔들린다.

**Independent Test**: 운영자가 소스 설정 파일을 수정한 뒤 수집기를 실행했을 때, 최근 7일 이내의 공식 항목이 정규화된 후보 목록으로 저장되면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 운영자가 공식 RSS, changelog, GitHub Releases API 기반 소스를 설정 파일에 등록해 두었고, **When** 수집 작업을 실행하면, **Then** 시스템은 각 소스에서 최근 7일 이내 항목만 읽어 공통 스키마로 정규화한다.
2. **Given** 동일한 소식이 RSS와 GitHub Releases API에서 중복으로 들어오고, **When** 후보 정리 단계가 실행되면, **Then** 시스템은 중복 항목을 하나로 묶고 canonical URL과 대표 제목을 남긴다.
3. **Given** 일부 소스가 일시적으로 응답하지 않거나 피드 형식이 흔들리고, **When** 수집 작업을 실행하면, **Then** 시스템은 전체 실행을 중단하지 않고 실패한 소스를 로그에 남긴다.

---

### User Story 2 - 주간 다이제스트 초안을 한국어로 생성한다 (Priority: P1)

블로그 운영자는 후보 항목이 많더라도, 주간 가치가 높은 항목만 압축한 한국어 초안 글을 자동으로 받아야 한다.

**Why this priority**: 수집 결과를 사람이 그대로 읽는 것은 비용이 크다. 운영자가 실제로 얻고 싶은 결과는 발행 가능한 블로그 초안이다.

**Independent Test**: 정규화된 후보 목록을 입력으로 넣었을 때, 시스템이 `_posts/`에 Jekyll front matter를 포함한 주간 다이제스트 초안을 생성하고 각 항목에 원문 링크를 남기면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 최근 7일 후보가 20개 이상 수집되었고, **When** 초안 생성기를 실행하면, **Then** 시스템은 카테고리 quota와 제외 규칙을 적용해 8~12개의 핵심 항목만 선택한다.
2. **Given** 선택된 항목에 릴리즈, 보안, deprecation, 제품 업데이트가 섞여 있고, **When** 한국어 요약을 생성하면, **Then** 각 항목은 "무슨 변화인가", "왜 중요한가", "실무 영향", "원문 링크"를 포함해 정리된다.
3. **Given** 생성된 초안이 저장될 날짜가 이번 주 다이제스트의 발행일이고, **When** 파일을 작성하면, **Then** 시스템은 `_posts/YYYY-MM-DD-weekly-tech-roundup.md` 형식의 파일명을 사용하고 Jekyll 빌드가 가능한 Markdown을 만든다.

### User Story 3 - 주간 실행 결과를 PR로 검토하고 배포한다 (Priority: P2)

블로그 운영자는 자동 생성 품질이 아직 완전히 검증되지 않은 동안, 초안이 바로 공개되기보다 PR로 올라와 사람이 보고 merge할 수 있어야 한다.

**Why this priority**: 초기 운영 단계에서 가장 큰 위험은 품질이 낮은 글이 자동 발행되는 것이다. PR 검토 단계를 두면 자동화의 이점과 편집 통제를 함께 가져갈 수 있다.

**Independent Test**: 스케줄 또는 수동 실행으로 초안이 생성된 뒤, CI가 Jekyll 빌드를 통과하는 PR을 만들고 운영자가 merge하면 Pages 배포가 이어지면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 주간 스케줄이 도래했고 실행에 필요한 자격 증명이 설정되어 있으며, **When** workflow가 실행되면, **Then** 시스템은 수집부터 초안 생성까지 수행한 뒤 결과를 PR 또는 별도 브랜치 커밋으로 올린다.
2. **Given** 생성된 초안이 Jekyll 문법 오류를 포함하고, **When** 품질 검증 단계가 실행되면, **Then** 시스템은 PR 생성 또는 merge 진행 전에 빌드 실패를 표시한다.
3. **Given** 운영자가 PR을 검토 후 merge하고, **When** `main` 브랜치에 반영되면, **Then** 기존 GitHub Pages 배포 workflow가 사이트를 재배포한다.

### User Story 4 - 실패 원인을 추적하고 소스 구성을 점진적으로 개선한다 (Priority: P3)

블로그 운영자는 어떤 소스가 자주 깨지는지, 이번 주에 왜 항목 수가 부족했는지, 어떤 규칙 때문에 제외되었는지를 추적할 수 있어야 한다.

**Why this priority**: 소스 수가 늘어나면 운영 실패의 대부분은 수집 품질 저하와 필터링 기준 불명확성에서 발생한다. 초기부터 흔적을 남겨야 자동화를 길게 운영할 수 있다.

**Independent Test**: 실행 후 남는 메타데이터와 로그만 보고도 "어떤 소스가 실패했는지, 몇 개가 제외되었는지, 최종 채택된 항목이 무엇인지"를 운영자가 확인할 수 있으면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 수집 소스 중 일부가 HTTP 오류를 반환하고, **When** workflow가 끝나면, **Then** 실행 결과에는 실패 소스, 오류 요약, 재시도 여부가 남아 있다.
2. **Given** 후보가 quota 적용과 제외 규칙으로 걸러지고, **When** 실행 결과를 확인하면, **Then** 최종 채택 전후의 항목 수와 주요 제외 이유를 볼 수 있다.
3. **Given** 한 주에 게시 가치가 낮은 항목만 수집되고, **When** 초안 생성 기준을 통과하지 못하면, **Then** 시스템은 빈 글을 발행하지 않고 사람 검토가 필요한 상태로 남긴다.

### Edge Cases

- 특정 소스가 RSS를 제공하지 않고 listing page만 제공할 때, HTML 구조가 바뀌어도 전체 파이프라인이 멈추지 않아야 한다.
- 동일 제품의 블로그 글과 릴리즈 API가 같은 내용을 다른 제목으로 제공할 수 있다.
- 한 주에 대형 이벤트가 많아 특정 카테고리의 후보가 몰릴 수 있다.
- 최근 7일 내 공식 소식이 너무 적어 quota를 채우지 못할 수 있다.
- LLM 요약 결과가 너무 마케팅 문구에 가깝거나 원문 취지를 왜곡할 수 있다.

## Requirements

### Functional Requirements

- **FR-001**: 시스템은 공식 RSS, 공식 changelog listing, 공식 GitHub Releases API를 입력 소스로 지원해야 한다.
- **FR-002**: 시스템은 소스별 메타데이터를 저장소 내 설정 파일로 관리해야 한다.
- **FR-003**: 시스템은 기본적으로 최근 7일 이내 항목만 수집 대상으로 삼아야 한다.
- **FR-004**: 시스템은 수집 항목을 공통 스키마로 정규화해야 한다.
- **FR-005**: 시스템은 제목, URL, 발행일, 소스 식별자를 기준으로 중복 항목을 제거하거나 병합해야 한다.
- **FR-006**: 시스템은 포함 규칙과 제외 규칙을 적용해 게시 가치가 높은 후보만 남겨야 한다.
- **FR-007**: 시스템은 카테고리 quota를 적용해 특정 제품군에 편중되지 않은 주간 다이제스트를 만들어야 한다.
- **FR-008**: 시스템은 선택된 항목에 대해 한국어 요약을 생성하고 원문 링크를 반드시 포함해야 한다.
- **FR-009**: 시스템은 Jekyll이 바로 빌드할 수 있는 Markdown 포스트 파일을 `_posts/` 아래에 생성해야 한다.
- **FR-010**: 시스템은 초기 운영 단계에서 기본 발행 모드를 "자동 PR 생성"으로 둬야 하며, 기본값으로 직접 `main`에 push하지 않아야 한다.
- **FR-011**: 시스템은 GitHub Actions 스케줄 실행과 수동 실행을 모두 지원해야 한다.
- **FR-012**: 시스템은 실행 중 실패한 소스와 제외 통계를 로그 또는 결과 요약 파일에 남겨야 한다.
- **FR-013**: 시스템은 유효한 초안을 만들지 못한 경우 빈 글을 발행하지 않아야 한다.
- **FR-014**: 시스템은 생성된 포스트에 날짜, 카테고리, 태그, excerpt를 포함한 front matter를 작성해야 한다.
- **FR-015**: 시스템은 현재 저장소 안에서 수집, 초안 생성, PR 생성, Pages 배포 흐름이 이어지도록 설계되어야 한다.

### Key Entities

- **SourceProfile**: 공식 소스 하나를 나타내는 설정 단위. 소스 이름, 유형, URL/API 엔드포인트, 카테고리, 신뢰도, 활성화 여부를 가진다.
- **CollectedEntry**: 수집 직후의 원시 항목을 정규화한 데이터. 제목, 원문 URL, 발행 시각, 요약 원문, 소스 식별자, 콘텐츠 유형을 가진다.
- **WeeklyCandidate**: 중복 제거와 규칙 적용 후 남은 후보 항목. 카테고리, 우선도 점수, 제외 사유, 채택 여부를 가진다.
- **WeeklyDigestDraft**: 최종 블로그 초안 단위. 발행 주차, 제목, front matter, 선택 항목 목록, Markdown 본문을 가진다.
- **PublishRun**: 주간 실행 한 번의 메타데이터. 실행 시각, 수집 성공/실패 수, 최종 채택 수, 생성 파일 경로, PR 상태를 가진다.

## Success Criteria

### Measurable Outcomes

- **SC-001**: 주간 실행 한 번으로 공식 소스 15개 이상에서 후보를 수집할 수 있다.
- **SC-002**: 정상 주간에는 최종 다이제스트에 8개 이상의 항목이 포함된다.
- **SC-003**: 생성된 초안은 추가 수작업 없이 `./scripts/blog-build.sh`를 통과한다.
- **SC-004**: 운영자는 주간 발행 초안을 15분 이내 검토 후 merge 또는 보류 결정을 내릴 수 있다.
- **SC-005**: 빈 글 발행, 원문 링크 누락, 중복 항목 중복 게시가 기본 경로에서 발생하지 않는다.

## Assumptions

- 초기 버전은 현재 Jekyll + GitHub Pages 저장소 안에서 동작하며 별도 저장소를 두지 않는다.
- 초기 버전의 자동화 목표는 "완전 자동 공개"가 아니라 "주간 초안 PR 생성"이다.
- 소스 범위는 AI/플랫폼, 웹/프론트엔드, 클라우드/인프라, 핵심 OSS 릴리즈를 우선한다.
- 소스 평가는 공식성, 변경 신호 밀도, 기계 수집 안정성을 우선 기준으로 삼는다.
- 장기적으로는 캐시와 실행 이력 저장소가 분리될 수 있지만, v1에서는 파일 기반 메타데이터와 Actions 로그로 시작한다.
